# How to Save the World — Governance

How to Save the World is governed by principles of decentralization, transparency, and community sovereignty.

**Note:** FOXHOLE Protocol is a canonical bonding protocol inside the NeuroVerse, not the name of this platform. This PWA implements the FOXHOLE bonding moment as part of onboarding; the full protocol definition lives in the NeuroVerse Canon repository.

---

## 🌟 Core Governance Principles

### 1. Transparency

- All code is open-source and auditable on GitHub
- All decisions are documented publicly
- All contributions are reviewed transparently

### 2. Inclusivity

- Everyone can contribute, regardless of background or experience
- All voices are heard and considered
- No gatekeeping or exclusionary practices

### 3. Sovereignty

- Users own their data, identity, and cognitive evolution
- No central authority controls operator progress or reflections
- Fork-friendly, DAO-compatible architecture

### 4. Remixability

- How to Save the World can be forked, modified, and remixed
- Content is licensed under CC BY-NC-SA 4.0 (remix-friendly)
- Application code is MIT licensed (fork-friendly)

### 5. Meritocracy

- Contributions are evaluated on merit, not politics
- Quality and impact determine influence
- No hierarchy, no favoritism

## 👥 Decision-Making Structure

### Lead Maintainer

**Kirsten Bischoff** serves as Lead Maintainer and has final decision authority on:

- Core curriculum changes (90-lesson structure)
- Archetype system modifications
- Echelon personality updates
- NeuroVerse OS integration changes

### Community Contributors

**All contributors** have decision authority on:

- UI/UX improvements
- Bug fixes
- Performance optimizations
- ACE Box content additions
- Translations
- Documentation improvements

### Decision Process

1. **Propose** — Create a GitHub Issue or Discussion
2. **Discuss** — Community reviews and provides feedback
3. **Iterate** — Author refines the proposal based on feedback
4. **Decide** — Lead Maintainer approves or requests changes
5. **Implement** — Contributor submits a Pull Request
6. **Review** — Code review and testing
7. **Merge** — PR merged into main branch
